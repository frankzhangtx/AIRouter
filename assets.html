Project Relationship Guide

How files in
cctest fit together

This Android project is a single Gradle app module. Kotlin screens, XML layouts, navigation, manifest declarations, resources, tests, and generated binding classes all play different roles. The page below shows the important connections so you can understand the project as a system, not just a folder tree.

Module Shape 1 app `settings.gradle.kts` includes only `:app`, so everything ships from one Android application module.
Entry Point MainActivity The launcher activity hosts the navigation graph and starts the app experience.
UI Style View Binding `app/build.gradle.kts` enables view binding, so XML layouts generate strongly typed binding classes.
Navigation Fragments + Activities The app mixes fragment-based flow inside `MainActivity` with standalone demo/detail activities.

Project layers

The repository is easiest to read in layers. Build files define the module. The manifest declares app entry points. Kotlin classes control behavior. XML resources describe layouts, navigation, strings, themes, and drawables. Gradle then generates binding classes and compiled output from those source assets.

Build and module setup

`settings.gradle.kts`, root `build.gradle.kts`, and `app/build.gradle.kts` decide what module exists and which Android features are enabled.

App declaration

`AndroidManifest.xml` registers `MainActivity`, `PersonalInfoDetailActivity`, `HouseDashboardActivity`, and `InsuranceMallActivity`.

UI behavior

Kotlin files under `app/src/main/java/com/example/cctest/` inflate bindings, wire buttons, navigate, and launch activities.

UI assets

Layouts, strings, colors, themes, drawables, menus, and the navigation graph live under `app/src/main/res/`.

Core relationship map

The most important dependency chain in this repo looks like this:

Gradle config enables Android + Kotlin + View Binding.

Manifest tells Android which activities exist and which one launches first.

MainActivity loads `activity_main.xml`, attaches a NavController, and uses `nav_graph.xml`.

Fragments load their own XML layouts and navigate between destinations declared in the graph.

Standalone activities load separate layouts directly and do not depend on the navigation graph.

Resources feed both Kotlin and XML through `R.*` references and generated binding classes.

Source files point inward to generated outputs. You edit Kotlin/XML/resources; Gradle produces the binding and compiled artifacts in `app/build/`.

Screen flow in this project

This is the concrete UI flow defined by the current code. Fragment navigation stays inside `MainActivity`, while two activity demos and one detail screen branch out as separate destinations.

Main launcher

`AndroidManifest.xml` marks `MainActivity` as the launcher. `MainActivity.kt` inflates `ActivityMainBinding`, which is generated from `activity_main.xml`.

app/src/main/AndroidManifest.xml app/src/main/java/com/example/cctest/MainActivity.kt app/src/main/res/layout/activity_main.xml

Fragment navigation host

`MainActivity` finds `nav_host_fragment_content_main` and connects it to `nav_graph.xml`, which starts at `FirstFragment`.

app/src/main/res/navigation/nav_graph.xml app/src/main/res/layout/content_main.xml FirstFragment.kt

Second fragment branches

`SecondFragment` navigates back to `FirstFragment`, opens `PersonalInfoFragment`, opens `PersonalInfoListFragment`, or launches two standalone activities with intents.

SecondFragment.kt fragment_second.xml HouseDashboardActivity.kt InsuranceMallActivity.kt

List to detail activity

`PersonalInfoListFragment` builds mock data, shows it in a `ListView`, then starts `PersonalInfoDetailActivity` with extras such as name, age, job, city, and phone.

PersonalInfoListFragment.kt PersonalInfoDetailActivity.kt activity_personal_info_detail.xml

What each file category is responsible for

Android projects look crowded because one feature usually spans several file types. That split is deliberate: behavior, structure, content, styling, and registration each live in different places.

Kotlin source

Owns behavior: button clicks, navigation, intents, toasts, data formatting, and lifecycle handling.

Layout XML

Owns screen structure: toolbars, text fields, lists, buttons, nav hosts, and visual arrangement.

Value resources

Own shared text and styling primitives: `strings.xml`, `colors.xml`, `themes.xml`, and dimensions.

Generated/build output

Owns derived artifacts under `app/build/`, including binding classes, compiled classes, dex files, and reports.

File catalog with examples from this repo

These examples show the real cross-file patterns already present in `cctest`.

Build and wiring

  • `settings.gradle.kts` names the root project and includes `:app`.
  • `app/build.gradle.kts` sets SDK levels, dependencies, and `viewBinding = true`.
  • `AndroidManifest.xml` is the app-level registry of activities and launcher settings.

Feature screens

  • `MainActivity.kt` + `activity_main.xml` + `content_main.xml` form the fragment host shell.
  • `PersonalInfoFragment.kt` + `fragment_personal_info.xml` make an input-and-summary screen.
  • `HouseDashboardActivity.kt` + `activity_house_dashboard.xml` form a standalone activity flow.

Shared resources

  • `values/strings.xml` supplies button labels, summaries, titles, and placeholder text.
  • `drawable/` contains icon and background assets used by the insurance and house screens.
  • `navigation/nav_graph.xml` centralizes fragment destinations and actions.

Why View Binding matters here

This project uses view binding instead of `findViewById`. That means each layout file creates a generated binding class, and Kotlin code inflates or references views through that type-safe class.

Example

`activity_main.xml` generates `ActivityMainBinding`, which `MainActivity.kt` uses to access `toolbar` and `fab` safely.

Fragment pattern

`FragmentFirstBinding`, `FragmentSecondBinding`, and others are created from fragment layouts and cleared in `onDestroyView`.

Tests and generated output

Not everything in the repo is hand-written source. Some directories exist to verify code, while others are products of the Android build.

Tests

`app/src/test/` holds local JVM tests and `app/src/androidTest/` holds device or emulator tests.

Build output

`app/build/` contains compiled classes, generated bindings, dex archives, and merge reports. These reflect source relationships but are not the source of truth.

The simplest mental model is: Gradle defines the app, the manifest exposes entry points, Kotlin controls behavior, XML resources describe UI and shared values, and `app/build/` is generated from all of that source. If you add a new feature, you will usually touch one Kotlin class, one or more XML resources, and sometimes the manifest or navigation graph.
