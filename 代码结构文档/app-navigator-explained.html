AppNavigator：把路由意图翻译成真正的页面跳转

这个类本身不做“意图判断”，它只负责把已经规划好的 RouteTarget 落成 Android 导航动作。 它同时处理两类目标：NavController 里的 Fragment 跳转，以及通过 startActivity 打开的独立 Activity。 你特别问到的 navController.currentDestination?.id == destinationId，本质上是一个“避免重复导航”的保护分支。

目标类：AppNavigator
源码：app/src/main/java/com/example/cctest/navigation/AppNavigator.kt
入口调用：MainActivity.onCreate() 收集 UiEffect.NavigateTo
重点问题：当前页面已是目标页时为什么直接返回

阅读导引

先抓职责，再看特殊分支，最后再回到业务场景。

1. 它是一个“执行器”，不是“决策器”

要跳哪里，已经在上游由 RoutingSessionViewModel、TargetPlanner、JourneyPlanner 决定好了。 AppNavigator 只做最后一步：把不同类型的 RouteTarget 映射成 navigate() 或 startActivity()。

2. Fragment 和 Activity 走两条不同通道

NavTarget、WorkflowTarget、ListTarget 统一走 NavController； DetailTarget 和 HouseDashboardTarget 直接打开 Activity。

3. 重复目标页时不再发起一次导航

currentDestination?.id == destinationId 说明“用户已经在目标 Fragment 上”。 这时继续 navigate() 的业务价值很低，反而可能制造重复栈、重复页面实例，或者造成多余的生命周期波动。

4. 列表页是唯一被特殊照顾的重复导航场景

如果目标是 PersonalInfoListFragment，并且有新的 focusRequest 参数，代码会把参数合并到当前回退栈条目的 arguments 里，表达“我不切页，但我想刷新当前列表的定位条件”。

核心代码切片

下面是和业务问题最相关的代码，注释是说明文档里的阅读注释，不是源码里的真实注释。

// AppNavigator 只有两个依赖：一个 Context，一个 NavController
class AppNavigator(
    private val context: Context,
    private val navController: NavController
) {
    fun navigate(target: RouteTarget) {
        when (target) {
            // 普通 Fragment 跳转，直接用 target 给出的 destinationId + args
            is RouteTarget.NavTarget -> navigateToDestination(target.destinationId, target.args)

            // 工作流目标本质上也是进入某个 Fragment，只是目的地来自 workflow 的入口 step
            is RouteTarget.WorkflowTarget -> navigateToDestination(
                destinationId = target.entryDestinationId,
                args = null
            )

            // 列表目标固定进入 PersonalInfoListFragment，并带上 focusRequest
            is RouteTarget.ListTarget -> navigateToDestination(
                destinationId = R.id.PersonalInfoListFragment,
                args = target.focusRequest.toBundle()
            )

            // 详情和家居看板不走 NavController，而是直接起 Activity
            is RouteTarget.DetailTarget -> {
                context.startActivity(PersonalInfoDetailActivity.createIntent(context, target.recordRef))
            }
            is RouteTarget.HouseDashboardTarget -> {
                context.startActivity(HouseDashboardActivity.createIntent(context, target.launchSpec))
            }
        }
    }

    private fun navigateToDestination(destinationId: Int, args: Bundle?) {
        // 关键保护：如果当前页已经就是目标页，就不要再 navigate 一次
        if (navController.currentDestination?.id == destinationId) {
            // 但列表页比较特殊：已经在列表上时，仍可能需要更新“定位到第几条 / 定位到谁”的参数
            if (destinationId == R.id.PersonalInfoListFragment && args != null) {
                navController.currentBackStackEntry?.arguments?.putAll(args)
            }
            return
        }

        // 只有当前页不是目标页时，才真正执行 Fragment 导航
        navController.navigate(destinationId, args)
    }
}
一句话概括： AppNavigator 做的是“把目标落地”，而不是“猜用户想去哪”。因此它的分支逻辑非常像一个适配器： 输入是统一抽象的 RouteTarget，输出是不同导航基础设施的调用。

协作关系

这几个类一起决定它为什么要这样写。

MainActivity 在 onCreate() 里创建 AppNavigator，并在收集到 UiEffect.NavigateTo 时调用 appNavigator.navigate(effect.target)。也就是说，它是界面层统一的导航出口。
RouteTarget 这是导航目标的统一协议。AppNavigator 之所以能保持很薄，是因为上游已经把目标整理成 NavTarget、WorkflowTarget、ListTarget、DetailTarget、HouseDashboardTarget。
JourneyExecutor 执行“多步路径”时，它也会间接调用 AppNavigator。如果当前目的地已经到了，awaitDestination() 会直接返回， 这和 AppNavigator 的“避免重复导航”是配套设计。
PersonalInfoListFragment 列表页首次进入时会从 arguments 里读 ListFocusRequest；而多步 Journey 执行时，它又会监听 savedStateHandle 上的列表定位事件。这解释了为什么列表页会成为唯一的重复导航特例。

`currentDestination?.id == destinationId` 的业务场景

这是你最关心的点，这里单独展开。

场景一：工作流或普通导航重复命中当前页

比如上游又发来一次“进入个人信息表单”的目标，但用户其实已经停在这个表单 Fragment 上。 这时没有必要再 navigate() 一次，因为页面已经到了。直接返回可以避免重复入栈和无意义的页面重建。

场景二：用户已经在个人信息列表页，又收到新的列表定位请求

这是最像业务需求的场景。比如用户先打开列表页，接着又输入“定位到第 12 条”或“找张三”。 目标页面没变，仍然是列表页，但页面内部的“聚焦对象”变了。所以代码不重新跳页，只尝试更新当前条目的参数。

为什么不直接重新 navigate 到同一个 Fragment

因为重新导航到同一个 destination 往往会带来重复实例、回退栈膨胀，或者至少让用户感知到一次没有必要的切页。 从这个类的意图看，作者想把“换页”和“同页内刷新状态”区分开。

需要注意的实现细节： 在当前代码里，PersonalInfoListFragment 只在首次创建时通过 ListFocusRequest.fromBundle(arguments) 读取一次 arguments。如果页面已经显示出来，再执行 currentBackStackEntry?.arguments?.putAll(args)， Fragment 本身并没有显式监听这类 arguments 变化。 所以这段分支表达的是一个明确的业务意图：“已经在列表页时不要重进页面，只更新目标参数”； 但从现有实现看，真正稳定触发列表内刷新的是 JourneyExecutor 里基于 savedStateHandle 的 KEY_LIST_FOCUS_REQUEST 事件通道，而不是这里的 arguments.putAll()。
如果你把这段逻辑翻译成产品语言，它大致等于： “同一个页面就不要反复打开；只有当目标页真的变了才切页。若仍在同一列表页，只更新列表当前该聚焦哪一条数据。”
