EntryActionRegistry 类说明与 resolvePath() 流程图

本页解释 app/src/main/java/com/example/cctest/navigation/EntryActionRegistry.kt 中这个类的职责、内部数据和两个公开方法。重点会放在 resolvePath(from, to)，因为它决定“当前页面怎样一步步跳到目标页面”， 并且它在实际运行时会被 JourneyPlanner.resolvePathSteps() 调用，用来生成一串可执行的导航动作。

类定位 页面跳转动作表 把“从哪个页面出发，可以执行哪些入口动作”集中注册。
对外方法 2 个 actionsFrom() 负责查表，resolvePath() 负责找路径。
算法核心 广度优先搜索 按层扩展可达页面，优先找到跳转步数更少的路径。
调用位置 JourneyPlanner 由 resolvePathSteps() 调它，把动作转换成实际的 JourneyStep。

先看这类到底在管什么

EntryActionRegistry 本身并不直接执行页面跳转。 它只维护一份“动作图谱”：每个 EntryAction 都描述一个从某页面到另一页面的可用入口动作。 之后其他模块再把这些动作翻译成真正的导航步骤。

source: EntryActionRegistry.kt:19-114

内部数据 1：actions

这是完整动作列表，当前代码里一共注册了 10 条动作。 每条动作包含 4 个字段：key、from、to、displayLabel。

  • key：动作类型，用于后续映射成具体 JourneyStep
  • from：动作允许从哪个页面发起
  • to：执行这个动作后会到达哪个页面
  • displayLabel：给 UI 或日志看的动作文案

内部数据 2：actionsBySource

这是对 actions 做 groupBy { it.from } 之后的结果。 它把“按顺序扫描全表”变成了“按起点页面直接取动作列表”，是 actionsFrom() 和 resolvePath() 的底层索引。

  • key 是起点页面 AppScreen
  • value 是从该页面可以发出的全部 EntryAction
  • 这样查询时不需要每次遍历全量动作表

可以把这个类理解成一张“有向图”的注册中心： 页面是节点，EntryAction 是有方向的边。 resolvePath() 做的事，本质上就是在这张图里找一条从起点到终点的可达路径。

这个类里的各个方法

严格来说，这个类对外只有两个公开方法。 真正需要重点理解的是 resolvePath()， 但它依赖 actionsFrom() 提供邻接节点，所以两者要连起来看。

fun actionsFrom(screen: AppScreen): List<EntryAction>

这个方法的职责非常单一：返回“从当前页面出发，能执行哪些动作”。 它本质上是对 actionsBySource[screen].orEmpty() 的包装。

  • 输入：一个起点页面 screen
  • 输出：该页面对应的动作列表；如果没有配置，则返回空列表
  • 价值：把“查动作表”的细节收口，不让外部直接碰内部索引结构
例如传入 AppScreen.SECOND，返回值里会有三个动作： OPEN_PERSONAL_INFO_LIST、 ENTER_PERSONAL_INFO_WORKFLOW、 OPEN_HOUSE_DASHBOARD。

fun resolvePath(from: AppScreen, to: AppScreen): List<EntryAction>

这个方法负责从注册好的动作图谱里，找出一条从 from 到 to 的动作链。 返回值不是页面列表，而是按顺序可执行的 EntryAction 列表。

  • 如果起点和终点相同，直接返回空列表，表示不需要跳
  • 如果能找到路径，返回一条按顺序排列的动作链
  • 如果找不到路径，也返回空列表，交给上层走兜底逻辑
它使用的是广度优先搜索，而不是深度优先搜索。 这意味着它会优先找到“跳转步数更少”的那条路径。

resolvePath() 的执行过程，按代码顺序拆开讲

下面这 6 步基本就是 EntryActionRegistry.kt:89-112 的真实执行流程，只是换成了便于阅读的中文。

1

先处理最简单的场景：起点和终点相同

如果 from == to，方法立刻返回 emptyList()。 这里的含义不是“没找到”，而是“本来就已经在目标页，不需要任何动作”。

2

初始化搜索状态

visited 初始包含起点页面，用来防止页面重复入队； queue 里先放入一项：(from, emptyList())， 表示“当前站在起点，还没有走过任何动作”。

3

循环从队列头部取出一个待扩展页面

每次 removeFirst() 取到的都是当前层最早进入队列的页面， 这就是广度优先搜索的关键：先扩展所有一步可达，再扩展两步可达，再扩展三步可达。

4

读取当前页面能发出的所有动作

代码通过 actionsFrom(currentScreen) 拿到邻接边。 对每个动作来说，action.to 就是下一跳页面。

5

过滤重复页面，构造下一条路径

如果 visited.add(action.to) 失败，说明这个页面之前已经探索过，直接 continue。 否则就把当前路径和这个动作拼起来，得到 nextPath = path + action。

6

命中目标则返回，否则继续入队搜索

如果 action.to == to，说明已经找到一条可达路径，立即返回 nextPath； 否则把 (action.to, nextPath) 放回队列尾部，后续继续扩展。 全部搜索完仍找不到，就返回空列表。

为什么说它偏向“最短路径”？ 因为 BFS 是按层推进的： 先检查 1 步能到的页面，再检查 2 步能到的页面。 所以第一次命中目标时，得到的通常就是“动作数最少”的那条路径。

把它放回本项目里看，会更容易理解

当前主调用点在 app/src/main/java/com/example/cctest/routing/workflow/JourneyPlanner.kt:126-144 的 resolvePathSteps()。 它先让 EntryActionRegistry 算路径，再把每个动作映射成真正可执行的 JourneyStep。

1. 已经在目标页

如果 entryScreen == targetScreen， JourneyPlanner.resolvePathSteps() 会先直接返回空列表， 这和 resolvePath() 的第一层判断一致。

2. 找到动作链

一旦 resolvePath() 返回非空动作列表， JourneyPlanner 就会把每个 EntryAction 交给 actionToSteps()，转换成导航步骤。

3. 没找到动作链

如果返回空列表，JourneyPlanner 会退回到调用方传入的 directStep，也就是一个直接导航的兜底方案。

所以这个注册表不是“唯一跳转方式”，而是“优先尝试的显式路径规划器”。 它擅长把一些必须先绕回中间页、再进入目标页的业务路径表达清楚； 找不到时，上层仍然保留直接跳转的兜底能力。

resolvePath() 的几个实际例子

这些例子都直接来自当前注册表，不是抽象演示。

例 1：从智能入口页进入个人信息工作流

输入：from = INTENT_ENTRY，to = PERSONAL_INFO_FORM

INTENT_ENTRY → RETURN_TO_SECOND → SECOND → ENTER_PERSONAL_INFO_WORKFLOW → PERSONAL_INFO_FORM

返回动作链会是 2 步： [RETURN_TO_SECOND, ENTER_PERSONAL_INFO_WORKFLOW]。

例 2：从第二页打开家居看板

输入：from = SECOND，to = HOUSE_DASHBOARD

SECOND → OPEN_HOUSE_DASHBOARD → HOUSE_DASHBOARD

这是 1 步直达，BFS 会第一层就命中，立即返回。

例 3：从列表页进入详情页

输入：from = PERSONAL_INFO_LIST，to = PERSONAL_INFO_DETAIL

PERSONAL_INFO_LIST → OPEN_FOCUSED_DETAIL → PERSONAL_INFO_DETAIL

当前图谱里这也是单步路径。 不过在现有调用链中，详情页更多是由 buildDetailJourney() 先聚焦列表后再打开。

例 4：起点和终点相同

输入：from = SECOND，to = SECOND

不搜索，不入队，不遍历动作，直接返回 []

这里的空列表代表“零步完成”，不是“找不到路”。

如果你传入一个当前图谱根本连不上的组合，例如某些页面之间没有任何可达通路， 那么 resolvePath() 会在队列清空后返回 emptyList()。 对当前工程来说，这通常意味着上层会退回到 directStep 兜底，而不是在这里抛异常。
