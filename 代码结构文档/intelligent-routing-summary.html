Journey-First Review

智能路由升级提案
从直达到回放

现在需要修正的不是“识别能力不够强”，而是“识别后执行得像瞬移”。新版方案保持 IntentParser、SlotNormalizer 和本地 planner， 但把执行模型升级为 RouteTarget + JourneyPlan + JourneyExecutor： 先确定最终目标，再规划像用户一样经过哪些页面，最后顺序执行。

默认体验 智能路由优先回放现有页面路径，而不是 deep link 直跳终点。
关键节点 SecondFragment 将从示例页升级为首页动作的显式中转页。
Parser 边界 模型仍只输出 UserGoal + Slots，不负责决定经过哪个 Fragment。
首期验收 “打开 Work 看板”必须先看到第二页，再进入看板并停在 WORK。
主执行模型
Journey 目标语义与页面路径分开建模，执行上做顺序回放。
规划拆分
2 层 TargetPlanner 决定最终目标，JourneyPlanner 决定中间步骤。
首期关键路径
3 条 看板、列表定位、详情查看都要可感知地经过中间页面。
实施顺序
4 段 先补可达关系和 Journey，再接真实 LLM。

本次评审真正要拍板的主流程

不是“识别完以后跳到哪里”，而是“识别完以后怎么像真实用户一样到达那里”。

Core Flow
Step 01

理解语义

Parser 识别用户意图、目标对象和歧义，但不决定 UI 动作。

Step 02

确定最终目标

TargetPlanner 把语义收敛成最终 RouteTarget。

Step 03

规划到达路径

JourneyPlanner 决定先去哪一页、再触发哪个动作。

Step 04

顺序执行 Journey

JourneyExecutor 逐步导航、聚焦、打开详情或启动 Activity。

Step 05

必要时停下

歧义时停在列表页并展示候选，不允许跳到用户无法解释的终态。

Journey 版最大的变化是：先解决“怎么去”，再谈“理解得多聪明”。 否则接了 LLM 以后，只会把“瞬移”做得更聪明，但产品体验仍然不自然。

为什么现在先改执行体验

当前真正割裂的是“手动入口”和“智能入口”的到达过程，而不是 parser 接口本身。

Why Now

用户更能建立心智

看到第二页、列表页这些中间节点后，用户更容易理解系统是怎么找到目标的。

回退更自然

详情歧义时停在列表页，比直接跳错详情或回首页更符合预期。

解释和重试更容易

Journey 拥有中间步骤语义，后续才能做“当前执行到哪一步了”的提示。

三条不能退的边界

执行方式升级后，解析和本地控制边界反而更重要。

Guardrails

Parser 只给语义

UserGoal + Slots 依旧是唯一对外输出，模型不能决定按钮点击顺序。

Journey 必须本地规划

目标语义、页面可达关系、列表歧义回退，都应由本地 registry 和 planner 负责。

页面关系必须显式声明

First -> Second -> List / Dashboard、List -> Detail 这些路径要成为一等契约。

新的骨架应该抓住什么

后续扩展能否平滑，取决于这些模型是否从第一天起就建正确。

Journey Spine

RouteTarget

保留最终业务目标语义，不和页面路径耦合。

JourneyPlan

记录 entry screen、steps、fallback policy 和 completion condition。

EntryActionRegistry

显式声明某页面能触发哪些“像用户操作”的入口动作。

JourneyExecutor

负责顺序执行、等待页面 ready、避免旋转屏后重复回放。

只有把“最终目标”和“到达路径”拆开，SecondFragment、PersonalInfoListFragment 这类中间节点才能被正确纳入智能路由主干。

建议实施节奏

顺序非常明确：先立住 Journey 主干，再接真实模型。反过来做会重复返工。

Roadmap
Stage 1

补可达关系

先回答“从当前页怎么去目标页”。

  • 声明 First -> Second
  • 声明 Second -> List / Dashboard
  • 声明 List -> Detail
Stage 2

引入 Journey 主干

从单点导航升级为顺序回放。

  • 新增 JourneyPlan 和 JourneyExecutor
  • ViewModel 增加 journey 状态
  • effect 改为执行 journey
Stage 3

跑通首期路径

优先稳定用户最容易感知的三条路径。

  • First -> Second -> Dashboard
  • First -> Second -> List -> Focus
  • First -> Second -> List -> Detail / Stop
Stage 4

最后再接真实 LLM

Journey 稳定后，再把模糊理解切到真实 provider。

  • 保持 parser 只输出语义
  • 保留本地路径规划控制权
  • 避免退回“理解后直跳终态”

评审结论

如果会上只保留四个判断，建议保留下面这四个。

Takeaways

先改执行体验

首要问题是“怎么去”，不是“识别得够不够聪明”。

Journey 要成为主干

RouteTarget 不够，需要显式建模中间步骤和执行顺序。

LLM 仍只做理解

模型负责识别意图，本地系统负责规划页面路径和动作回放。

SecondFragment 必须升格

它将成为首页进入列表、看板、工作流的重要 Journey 节点。
