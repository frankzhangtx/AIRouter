Technical Solution · Journey Routing

智能路由技术方案
Journey 执行版

这份技术方案不再把“文本 -> 终点页”当成唯一主线，而是改成“文本 -> 最终目标 -> JourneyPlan -> 顺序回放”。 重点不是让系统跳得更快，而是让系统在保持 parser 可切换的同时，严格遵循当前 App 已有页面路径和动作语义， 例如 FirstFragment -> SecondFragment -> HouseDashboardActivity。

默认行为 操作回放 优先回放已有页面路径，而不是首页直跳终态。
核心对象 JourneyPlan 最终目标和到达路径必须同时建模。
规划拆分 2 段 TargetPlanner 先定目标，JourneyPlanner 再定步骤。
LLM 接入时机 最后再接 先把 Journey 骨架立稳，再把模糊理解切给真实 provider。

技术主干

新版主干不是缩短路径，而是让“理解”和“执行”各自处在正确的层级里。

Technical Spine

1. Parser 继续只给语义

IntentParser 仍只输出 UserGoal + Slots，不返回 UI 动作或 Journey。

2. TargetPlanner 只定最终目标

把语义收敛成 RouteTarget、RecordRef、ListFocusRequest 等终态信息。

3. JourneyPlanner 负责“怎么去”

根据当前所在页面和可达关系，规划需要经过哪些页面、执行哪些动作。

4. JourneyExecutor 顺序回放

执行步骤、等待页面 ready、避免重复触发，并在必要时停在中间页。

分层架构图

执行模型改变后，架构边界应从“解析 -> 规划 -> 导航”升级为“解析 -> 目标规划 -> 路径规划 -> Journey 执行”。

核心协议

这些协议决定“理解”和“执行”之间有没有被正确隔离。

Contracts

ParseResult

仍然只包含受限业务语义，不允许穿透到 UI 细节层。

  • userGoal
  • slots
  • confidence
  • ambiguityReason

RouteTarget

只表示最终业务目标，不承载“先去哪一页”的路径信息。

  • WorkflowTarget
  • ListTarget
  • DetailTarget
  • HouseDashboardTarget

JourneyPlan

真正描述用户可感知执行路径的主对象。

  • entryScreen
  • steps
  • fallbackPolicy
  • completionCondition

执行约束

Journey 模型必须解决“像用户操作”，但不能退化成不受控的 UI 自动化。

Guardrails

模型不能返回 UI 拓扑

LLM 不允许输出“先去 SecondFragment”或“点击某个按钮”这类路径细节。

可达关系必须本地声明

First -> Second、Second -> List / Dashboard、List -> Detail 必须在本地 registry 里显式存在。

直达只能作为产品优化开关

默认策略仍然是 Journey 回放。只有产品明确要求时，才允许跳过中间页。

Journey 主流程

主流程从“识别后导航”升级成“识别后规划目标，再规划路径，再执行 Journey”。

Journey Flow
Step 01

自然语言输入

例如“打开 Work 看板”“查看第 12 条记录”“查看张雨桐的信息”。

→
Step 02

解析业务语义

规则优先，LLM 补强，但输出始终限定为 UserGoal + Slots。

→
Step 03

目标与路径规划

先得到最终 RouteTarget，再生成完整 JourneyPlan。

→
Step 04

顺序回放 Journey

进入中间页、聚焦列表、打开详情或启动 Activity，并在必要时停下。

Dashboard Journey

打开 Work 看板

用户应先看到第二页，再进入看板页，最后停在 WORK。

First Second Dashboard WORK
List Journey

查看第 12 条记录

先到第二页，再进列表页，最后在列表页执行聚焦和高亮。

First Second List Focus
Detail Journey

查看张雨桐的信息

默认先过列表页。唯一命中才打开详情，歧义时停在列表页等待确认。

First Second List Detail / Stop
Workflow Journey

补全个人资料

工作流也应复用 Journey 主干，避免让用户瞬移到难理解的中间步骤页。

First Second Form Review / Result

状态、副作用与页面就绪同步

Journey 能否稳定，取决于状态里有没有步骤索引，以及执行器能否在页面 ready 后再继续。

State & Effects

1. SessionState 增加 journey 维度

建议增加 activeJourneyId、currentJourneyStepIndex、journeyStatus。

2. UiEffect 改为执行 Journey

核心副作用从 NavigateTo(target) 升级为 ExecuteJourney(plan)。

3. 页面 ready 后再继续

JourneyExecutor 需要等待目标 Fragment / Activity 稳定后再执行下一步。

4. 歧义时允许暂停

Journey 状态应支持 PAUSED，让流程停在列表页而不是强行继续。

5. 旋转屏不能重复执行

动作本身仍然不能长期常驻状态，否则会在重建后再次打开看板或详情。

关键结论： Journey 不是“把多个导航动作串在一起”那么简单，它必须包含步骤索引、页面 ready 同步和暂停/失败处理能力。

可达关系与改造落点

代码改动应集中在页面可达关系、规划层和执行层，不把 Journey 细节扩散到每个页面各自判断。

Module Map

Reachability

描述页面和动作的可达关系。

  • DestinationContractRegistry.kt
  • EntryActionRegistry.kt
  • 声明 First -> Second、Second -> List / Dashboard

Planning

把最终目标和路径规划拆开。

  • TargetPlanner.kt
  • JourneyPlanner.kt
  • RouteTarget.kt、JourneyPlan.kt

State & Execution

负责会话状态、步骤索引和顺序回放。

  • RoutingSessionViewModel.kt
  • SessionState.kt
  • UiEffect.kt、JourneyExecutor.kt

Page Behavior

页面只负责暴露动作和承接回放。

  • SecondFragment.kt
  • PersonalInfoListFragment.kt
  • HouseDashboardActivity.kt、PersonalInfoDetailActivity.kt
页面角色变化： SecondFragment 不再只是示例页；它会成为首页进入列表、看板和工作流的重要 Journey 节点。 PersonalInfoListFragment 也不再只是终点页，而是详情页前的显式确认/聚焦页。

建议实施节奏

阶段顺序必须围绕 Journey 骨架展开，而不是先接真实 LLM 再回头修执行体验。

Phased Delivery
Stage 1

补页面可达关系

先把真实页面路径结构化。

  • 明确 First -> Second
  • 明确 Second -> List / Dashboard
  • 明确 List -> Detail
Stage 2

引入 Journey 核架

从单点导航升级为分步回放。

  • 新增 JourneyPlan、JourneyStep、JourneyExecutor
  • 增加 journey 状态和执行副作用
  • 打通页面 ready 同步逻辑
Stage 3

跑通三条关键路径

优先交付用户最容易感知的 Journey。

  • First -> Second -> Dashboard
  • First -> Second -> List -> Focus
  • First -> Second -> List -> Detail / Stop
Stage 4

最后再接真实 LLM

Journey 稳定后，再把模糊语义接到真实 provider。

  • 保持 parser 只输出语义
  • 通过 gateway 和 feature flag 接入 provider
  • 避免退回“理解后直跳终态”的旧模型
这版技术方案对应项目根目录的 add-intelligent-routing.md。 当前最重要的结论是：先立住 Journey 主干，再接真实 LLM。 如果顺序反过来，系统只会更擅长“猜到目标”，但仍然不会“像用户一样到达目标”。
