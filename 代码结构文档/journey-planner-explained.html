JourneyPlanner 类说明与执行流程图

JourneyPlanner 的核心职责不是识别用户想去哪里，而是在目标已经确定之后， 把 TargetPlan 翻译成一组可执行的 JourneyStep。 它把“当前页面在哪里”“目标类型是什么”“系统是否存在既定页面路径”这三类信息压缩成一份 JourneyPlan。

源码位置：app/src/main/java/com/example/cctest/routing/workflow/JourneyPlanner.kt
协作对象：RoutePlanner、TargetPlan、EntryActionRegistry、JourneyStep
适用场景：工作流、列表、详情、家居看板、普通 Nav 跳转

带中文注释的关键代码

这里不展开整个文件，而是保留最能体现控制流的片段：入口分发、路径解析、详情和列表的后续步骤追加。 

class JourneyPlanner(
    private val entryActionRegistry: EntryActionRegistry
) {
    fun plan(currentDestinationId: Int?, targetPlan: TargetPlan): JourneyPlan? {
        // 1. 如果 TargetPlanner 没有产出明确 target，这里不生成任何路线。
        val target = targetPlan.target ?: return null

        // 2. 把 Nav destinationId 归一到 AppScreen，未知页面统一按 SECOND 兜底。
        val entryScreen = normalizeEntryScreen(AppScreen.fromDestinationId(currentDestinationId))

        // 3. 不同 RouteTarget 对应不同的“补齐步骤”策略。
        val steps = when (target) {
            is RouteTarget.WorkflowTarget -> buildWorkflowJourney(entryScreen, targetPlan, target)
            is RouteTarget.ListTarget -> buildListJourney(entryScreen, targetPlan)
            is RouteTarget.DetailTarget -> buildDetailJourney(entryScreen, targetPlan, target)
            is RouteTarget.HouseDashboardTarget -> buildHouseDashboardJourney(entryScreen, targetPlan, target)
            is RouteTarget.NavTarget -> listOf(
                JourneyStep.NavigateToFragment(
                    destinationId = target.destinationId,
                    args = target.args,
                    displayLabel = target.displayName
                )
            )
        }

        // 4. 最后把 entry、target、steps 封成 JourneyPlan，交给后续执行器消费。
        return JourneyPlan(
            journeyId = UUID.randomUUID().toString(),
            entryScreen = entryScreen,
            target = target,
            steps = steps,
            fallbackPolicy = targetPlan.fallbackMessage ?: "stop_on_current_screen",
            completionCondition = target.displayName
        )
    }

    private fun buildListJourney(
        entryScreen: AppScreen,
        targetPlan: TargetPlan
    ): List<JourneyStep> {
        val steps = resolvePathSteps(
            entryScreen = entryScreen,
            targetScreen = AppScreen.PERSONAL_INFO_LIST,
            targetPlan = targetPlan,
            directStep = JourneyStep.NavigateToFragment(
                destinationId = R.id.PersonalInfoListFragment,
                displayLabel = "打开个人信息列表"
            )
        ).toMutableList()

        // 列表目标除了“到达列表页”，还可能追加“定位记录”“展示回退提示”。
        targetPlan.listFocusRequest
            ?.takeIf { it.hasLocator() }
            ?.let { steps += JourneyStep.FocusList(it) }
        targetPlan.fallbackMessage?.let { steps += JourneyStep.ShowFallbackMessage(it) }

        // 列表页是一个可停留终点，所以总会追加 StopAtCurrentScreen。
        steps += JourneyStep.StopAtCurrentScreen(
            reason = if (targetPlan.fallbackMessage != null) "fallback_list" else "list_ready"
        )
        return steps
    }

    private fun buildDetailJourney(
        entryScreen: AppScreen,
        targetPlan: TargetPlan,
        target: RouteTarget.DetailTarget
    ): List<JourneyStep> {
        val steps = resolvePathSteps(
            entryScreen = entryScreen,
            targetScreen = AppScreen.PERSONAL_INFO_LIST,
            targetPlan = targetPlan,
            directStep = JourneyStep.NavigateToFragment(
                destinationId = R.id.PersonalInfoListFragment,
                displayLabel = "打开个人信息列表"
            )
        ).toMutableList()

        // 详情页并不直接打开，而是先在列表中定位，再打开已聚焦记录的详情。
        val focusRequest = targetPlan.listFocusRequest ?: ListFocusRequest(
            recordId = target.recordRef.recordId,
            queryName = target.recordRef.displayName,
            queryPhone = target.recordRef.phone,
            queryCity = target.recordRef.city,
            matchMode = ListFocusRequest.MATCH_MODE_RECORD_ID
        )
        if (focusRequest.hasLocator()) {
            steps += JourneyStep.FocusList(focusRequest)
        }
        steps += JourneyStep.OpenFocusedRecordDetail(target.recordRef)
        return steps
    }

    private fun resolvePathSteps(
        entryScreen: AppScreen,
        targetScreen: AppScreen,
        targetPlan: TargetPlan,
        directStep: JourneyStep
    ): List<JourneyStep> {
        // 已经在目标屏幕上时，不重复导航，由上层补充后续动作即可。
        if (entryScreen == targetScreen) {
            return emptyList()
        }

        // 优先走系统登记好的页面路径，例如 INTENT_ENTRY -> SECOND -> HOUSE_DASHBOARD。
        val actions = entryActionRegistry.resolvePath(entryScreen, targetScreen)
        if (actions.isNotEmpty()) {
            return actions.flatMap { action ->
                actionToSteps(action, targetPlan)
            }
        }

        // 如果注册表没有路径，就直接执行 directStep 作为兜底。
        return listOf(directStep)
    }

    private fun actionToSteps(
        action: EntryAction,
        targetPlan: TargetPlan
    ): List<JourneyStep> {
        return when (action.key) {
            EntryActionKey.GO_TO_SECOND,
            EntryActionKey.RETURN_TO_SECOND -> listOf(
                JourneyStep.NavigateToFragment(
                    destinationId = R.id.SecondFragment,
                    displayLabel = action.displayLabel
                )
            )

            EntryActionKey.OPEN_PERSONAL_INFO_LIST -> listOf(
                JourneyStep.NavigateToFragment(
                    destinationId = R.id.PersonalInfoListFragment,
                    displayLabel = action.displayLabel
                )
            )

            // 这类 action 不只决定“去哪”，还会从 targetPlan 中取补充参数。
            EntryActionKey.ENTER_PERSONAL_INFO_WORKFLOW -> {
                val workflowTarget = targetPlan.target as? RouteTarget.WorkflowTarget
                listOf(
                    JourneyStep.NavigateToFragment(
                        destinationId = workflowTarget?.entryDestinationId ?: R.id.personalInfoFormStepFragment,
                        displayLabel = action.displayLabel
                    )
                )
            }

            EntryActionKey.OPEN_HOUSE_DASHBOARD -> {
                val houseTarget = targetPlan.target as? RouteTarget.HouseDashboardTarget ?: return emptyList()
                listOf(
                    JourneyStep.OpenHouseDashboard(
                        launchSpec = houseTarget.launchSpec,
                        displayLabel = action.displayLabel
                    )
                )
            }

            EntryActionKey.OPEN_FOCUSED_DETAIL -> {
                val detailTarget = targetPlan.target as? RouteTarget.DetailTarget ?: return emptyList()
                listOf(
                    JourneyStep.OpenFocusedRecordDetail(
                        recordRef = detailTarget.recordRef,
                        displayLabel = action.displayLabel
                    )
                )
            }
        }
    }
}
这个类最重要的建模点是：它把“路径查找”和“目标特有补充动作”分开了。 resolvePathSteps() 只负责决定怎样抵达目标屏幕， 真正的业务差异则在 buildListJourney() 和 buildDetailJourney() 这种方法里继续追加。

阅读抓手

先抓角色，再看分支，最后看兜底逻辑，理解这个类会更快。

它在整条链路中的位置

RoutePlanner.plan() 先调用 TargetPlanner 识别用户目标， 然后把得到的 TargetPlan 交给 JourneyPlanner.plan()。 所以它是“目标之后、执行之前”的编排层。

输入和输出

输入是当前 Nav 目的地 currentDestinationId 与包含业务上下文的 TargetPlan； 输出是 JourneyPlan，其中最关键的是 steps。

为什么要先走注册表路径

某些目标不能直接从任意入口打开，或者产品希望复用已有页面流。 EntryActionRegistry.resolvePath() 先尝试找一条显式路径，找不到时才直接跳。

详情目标为什么拆成两步

详情页依赖列表中的记录聚焦能力。代码先确保到达列表页，再发出 FocusList， 最后才执行 OpenFocusedRecordDetail，避免“没定位就开详情”。

列表目标为什么总要停留

列表类请求通常是给用户看一个结果集，或让用户自行确认歧义。 所以 buildListJourney() 无论是否有回退提示，都会追加 StopAtCurrentScreen。

一个值得注意的实现细节

fallbackPolicy 目前直接塞入了 fallbackMessage 或默认字符串， 从命名上看它更像“策略”，但现在承载了部分文案语义，后续如果扩展执行器，最好进一步拆分。

入口方法
plan(currentDestinationId, targetPlan)
关键依赖
EntryActionRegistry 负责路径搜索，TargetPlan 提供目标和补充参数。
核心输出
JourneyPlan.steps，由多个 JourneyStep 组成。
失败分支
targetPlan.target == null 时返回 null；部分 action 若 target 类型不匹配则返回空步骤。

聚焦分支：ENTER_PERSONAL_INFO_WORKFLOW

上面的总览里提到，这类 action 会从 targetPlan 读取运行时补充参数。 这一节把该分支单独拆开，说明为什么不能只靠 action.key，以及参数是如何生成并被消费的。

EntryAction 是路径注册表里的静态动作，只表达“从哪个屏幕到哪个屏幕、动作叫什么”。 当动作是“进入工作流”时，这些静态字段不足以决定最终该跳到哪个具体步骤页。 因此 JourneyPlanner 需要从 targetPlan.target 中取出 RouteTarget.WorkflowTarget.entryDestinationId，把抽象动作落成真实导航目标。

参数生成与消费链路

真正被这条分支消费的补充参数不是一组任意上下文字段，而是工作流入口步骤的 destinationId。 

// 片段 1：JourneyPlanner 在 actionToSteps() 中消费运行时入口参数。
EntryActionKey.ENTER_PERSONAL_INFO_WORKFLOW -> {
    // action.key 只能说明“进入工作流”，拿不到当前这次该进入哪个步骤页。
    val workflowTarget = targetPlan.target as? RouteTarget.WorkflowTarget
    listOf(
        JourneyStep.NavigateToFragment(
            // 真正被读取的补充参数就是 entryDestinationId。
            destinationId = workflowTarget?.entryDestinationId ?: R.id.personalInfoFormStepFragment,
            displayLabel = action.displayLabel
        )
    )
}

// 片段 2：TargetPlanner 在识别到 FillPersonalInfo 后，先启动工作流并拿到首个 step。
private fun planPersonalInfoWorkflow(...): TargetPlan {
    val workflow = workflowRegistry.personalInfoWorkflow()
    val step = workflowEngine.start(workflow)
    return TargetPlan(
        target = RouteTarget.WorkflowTarget(
            workflowId = workflow.workflowId,
            // 这里把“工作流入口页”写进 target，供 JourneyPlanner 后续消费。
            entryDestinationId = step.destinationId,
            displayName = contract.screen?.displayName ?: "个人信息工作流"
        ),
        workflowId = workflow.workflowId,
        currentStepId = step.stepId,
        ...
    )
}

// 片段 3：WorkflowEngine.start() 当前返回工作流的第一个 step。
class WorkflowEngine {
    fun start(definition: WorkflowDefinition): WorkflowStep {
        return definition.steps.first()
    }
}

// personalInfoWorkflow 的第一步现在刚好是表单页，但 JourneyPlanner 不应把它写死在逻辑上。
WorkflowDefinition(
    workflowId = PERSONAL_INFO_WORKFLOW_ID,
    steps = listOf(
        WorkflowStep(STEP_FORM, R.id.personalInfoFormStepFragment),
        WorkflowStep(STEP_REVIEW, R.id.reviewSubmitFragment),
        WorkflowStep(STEP_RESULT, R.id.resultFragment)
    )
)
这段设计把职责切得很清楚： TargetPlanner 和工作流层负责决定“这次入口在哪里”， JourneyPlanner 只负责把它翻译成 JourneyStep.NavigateToFragment。 这样一来，工作流入口发生变化时，不需要回头修改路径注册表的抽象 action。

为什么、取什么、做什么

这个分支可以压缩成三个问题来理解。

为什么必须从 targetPlan 取

因为 EntryAction 只有 key / from / to / displayLabel， 它是注册表里的静态路径定义。对于“进入工作流”这种动作，仅凭 action.key 只能知道意图，不能知道这次具体要落到哪个工作流步骤页。

实际取的是什么参数

当前这条分支真正读取的是 (targetPlan.target as? RouteTarget.WorkflowTarget)?.entryDestinationId。 也就是工作流入口页的 destinationId。这段代码并没有直接消费 workflowId、currentStepId 或 formFields。

这个参数的来源

它来自 TargetPlanner.planPersonalInfoWorkflow()。 该方法先调用 workflowEngine.start(workflow) 拿到首个 WorkflowStep，再把 step.destinationId 写进 RouteTarget.WorkflowTarget.entryDestinationId。

它最终起什么作用

它把抽象动作“进入个人信息工作流”落成了实际跳转 JourneyStep.NavigateToFragment(destinationId = ...)， 让执行器最终进入这次工作流真正的入口步骤页。

为什么不能只写死默认表单页

现在第一步刚好是 personalInfoFormStepFragment， 但如果以后工作流入口调整，或者需要按上下文从别的步骤恢复， 硬编码会让路径层直接依赖工作流细节。当前写法把这层变化隔离在目标规划阶段。

兜底逻辑说明了什么

?: R.id.personalInfoFormStepFragment 表示如果 targetPlan.target 不是 WorkflowTarget， 仍然回退到默认表单页，避免步骤完全生成失败。这是一个保守兜底，不是主路径。

消费参数
entryDestinationId
参数载体
RouteTarget.WorkflowTarget
参数来源
WorkflowEngine.start(workflow) 返回的 WorkflowStep.destinationId
最终效果
生成 JourneyStep.NavigateToFragment，把用户带到这次工作流真正的入口步骤页面。
