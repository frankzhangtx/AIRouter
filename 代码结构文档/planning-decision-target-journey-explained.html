PlanningDecision Analysis

在此工程里，
PlanningDecision 中的
targetPlan 和 journeyPlan
是怎么解析出来的

这份说明只聚焦一个问题：RoutePlanner.plan(...) 返回的 PlanningDecision，为什么会得到当前的 targetPlan 和 journeyPlan。核心结论是： targetPlan 由 TargetPlanner 根据 ParseResult 生成， journeyPlan 由 JourneyPlanner 根据 targetPlan 和当前页面生成。

返回结构 PlanningDecision 包含两个成员：targetPlan 与可空的 journeyPlan。
目标规划 TargetPlan 回答“最终要去哪、带什么业务上下文”。
路径规划 JourneyPlan 回答“从当前页如何一步步走到目标”。
调度者 RoutePlanner 自己不做复杂业务判断，只负责顺序组合两个 planner。

先看代码里真正的生成点

PlanningDecision 不是多处分散拼出来的，而是在 app/src/main/java/com/example/cctest/routing/workflow/RoutePlanner.kt 里一次性构造。

Core

生成顺序非常固定：先调用 targetPlanner.plan(result, currentFields) 得到 targetPlan，然后只有在 targetPlan.target != null 时， 才调用 journeyPlanner.plan(currentDestinationId, targetPlan) 生成 journeyPlan。如果目标都不存在，路径规划自然也不会发生。

RoutePlanner 主流程图

这一条流程线就是 RoutePlanner.plan() 的真实逻辑。

输入 ParseResult + currentDestinationId + currentFields
→
targetPlanner.plan(...) 先把用户意图和 slots 解释成业务目标。
→
得到 targetPlan 它可能包含 target、workflowId、recordRef、fallbackMessage 等。
→
targetPlan.target 是否为空 为空则 journeyPlan = null；不为空才继续。
→
journeyPlanner.plan(...) 把目标变成一串可执行步骤。

流程图一：PlanningDecision 进入点

从调用链上看，PlanningDecision 的生成起点在 RoutingSessionViewModel.submitUserInput()。

Entry
1

用户输入文本

例如“查看张雨桐详情”“填写李晨曦的个人信息”。

  • 先做空串校验
  • 写入 session state 的 rawInputText
2

生成 ParseRequest

会把当前页面和当前 workflow step 一起传给 parser。

  • currentDestinationId -> AppScreen
  • sessionSnapshot = currentStepId
3

得到 ParseResult

通过 parser 识别出 userGoal、slots、confidence。

  • 例如详情意图
  • 例如姓名、手机号、列表位置等 slot
4

标准化与校验

先经过 SlotNormalizer，再经过 ParseResultValidator。

  • 手机号只留数字
  • 详情意图必须带识别条件
5

调用 RoutePlanner

routePlanner.plan(normalizedResult, currentDestinationId, currentFields)

  • 返回 PlanningDecision
  • 随后写回 UI state 并发出 effect
也就是说，PlanningDecision 的直接输入不是原始文本，而是已经规范化、验证过的 ParseResult 与当前 UI 上下文。

流程图二：targetPlan 是如何解析出来的

TargetPlanner 是 “业务目标解析器”。它并不关心如何一步步导航， 它只关心根据用户意图最终应该落到哪类目标。

TargetPlan

TargetPlanner 决策流程

分支入口是 when (result.userGoal)。

FillPersonalInfo 表示用户要进入个人信息工作流。
→
读取 workflow 从 WorkflowRegistry.personalInfoWorkflow() 取得完整工作流定义。
→
取起始 step 用 WorkflowEngine.start() 拿到第一个 step，也就是 form 页。
→
生成 WorkflowTarget 写入 workflowId、currentStepId、formFields 与提示文案。
BrowsePersonalInfoList 表示用户要去列表页，可能还要定位某条记录。
→
resolveFromSlots() 用姓名 / 手机 / 城市 / 第 N 条 解析记录命中情况。
→
构造 ListFocusRequest 可能是唯一命中、位置命中、模糊查询，或空请求。
→
生成 ListTarget 写入 listFocusRequest、可选 recordRef 与列表提示文案。
OpenPersonalInfoDetail 表示用户要看详情页，但未必能直接命中唯一记录。
→
resolveFromSlots() 先判断详情目标是否唯一。
→
唯一命中？ 是：生成 DetailTarget。否：回退为 ListTarget。
→
附带回退语义 歧义或未命中时会写入 fallbackMessage。
OpenHouseDashboard / Unknown 分别是家居看板和无法识别请求。
→
看板分支 构造 HouseDashboardLaunchSpec 与 HouseDashboardTarget。
→
未知分支 仅返回 routeIntent = Unknown 和 fallbackMessage。
→
关键后果 Unknown 分支里 target = null，这会直接阻断后续 journey 生成。

targetPlan 每个字段来自哪里

target
由不同分支生成不同的 RouteTarget，例如 WorkflowTarget、ListTarget、DetailTarget、HouseDashboardTarget。
workflowId
只在填表工作流分支设置，来源是 WorkflowDefinition.workflowId。
currentStepId
只在工作流分支设置，来源是 WorkflowEngine.start() 返回的 step。
routeIntent
基本等于 ParseResult.userGoal，是该次规划的业务意图标签。
activeTargetLabel
多来自 DestinationContractRegistry 中 contract 的 screen 名称或 pageType。

剩余字段的来源

formFields
只在填表分支出现，由 currentFields.mergeFrom(result.slots.personalInfoFields) 生成。
recordRef
多来自 RecordResolution.toRecordRef()，用于详情页或列表定位上下文。
listFocusRequest
来自记录解析结果或 slot 自身，用于列表页聚焦。
infoMessage
表示正常引导提示，例如“已生成列表定位请求”“已进入个人信息工作流”。
fallbackMessage
表示回退提示，例如详情目标歧义时引导回列表确认。

详情分支为何最关键

它最能体现 targetPlan 不是“机械映射”，而是带有业务回退策略。

用户说“查看详情” 这只说明目标类型是 detail，并不说明具体是哪一条记录。
→
RecordResolver 查记录 按 recordId、phone、position、name、city 依次匹配。
→
唯一命中 生成 DetailTarget，后续可继续开详情。
→
歧义或未命中 不再坚持详情，回退生成 ListTarget。
→
结果 targetPlan 决定了后续 journey 是“打开详情链路”还是“停留列表确认链路”。

流程图三：journeyPlan 是如何解析出来的

JourneyPlanner 处理的是“路径问题”。此时目标已经由 targetPlan 决定，它只需要从当前页出发，拼出一串步骤。

JourneyPlan

JourneyPlanner 总流程

这部分直接对应 JourneyPlanner.plan() 方法。

读取 targetPlan.target 如果为空，直接返回 null，PlanningDecision 中的 journeyPlan 就为空。
→
currentDestinationId -> entryScreen 通过 AppScreen.fromDestinationId() 映射当前页面。
→
normalizeEntryScreen() 如果当前页识别不出来，就强制按 SECOND 作为默认起点。
→
按 target 类型分发 交给 buildWorkflowJourney、buildListJourney、buildDetailJourney 等。

WorkflowTarget 如何转成 journey

  1. 目标屏幕视为 PERSONAL_INFO_FORM
  2. 调用 resolvePathSteps(entryScreen, PERSONAL_INFO_FORM)
  3. 如果能找到路径，就按路径上的 action 转成 NavigateToFragment
  4. 如果找不到路径，则直接使用 entryDestinationId 形成一个直达步骤

ListTarget 如何转成 journey

  1. 先规划到 PERSONAL_INFO_LIST
  2. 若 targetPlan.listFocusRequest 存在且可定位，则追加 FocusList
  3. 若 fallbackMessage 存在，则追加 ShowFallbackMessage
  4. 最后固定追加 StopAtCurrentScreen

DetailTarget 如何转成 journey

  1. 先规划到 PERSONAL_INFO_LIST，不是直接进详情
  2. 再从 targetPlan.listFocusRequest 或 recordRef 补出定位信息
  3. 如果有定位条件，追加 FocusList
  4. 最后追加 OpenFocusedRecordDetail

JourneyPlanner 里的路径搜索图

底层依赖是 EntryActionRegistry.resolvePath() 的 BFS 搜索。

输入 from = entryScreen，to = targetScreen
→
actionsFrom(current) 取当前屏幕可走的所有边，例如 FIRST -> SECOND。
→
BFS 逐层扩展 遇到第一个命中 to 的路径就返回，天然是较短路径。
→
actionToSteps() 把每个 EntryAction 翻译为一个或多个 JourneyStep。
→
生成 JourneyPlan 填入 journeyId、entryScreen、steps、fallbackPolicy。

为什么 journeyPlan 依赖 targetPlan

  • 因为目标类型不同，补的步骤不同
  • DetailTarget 需要 recordRef 或 listFocusRequest
  • HouseDashboardTarget 需要 launchSpec
  • fallbackMessage 会直接影响 fallbackPolicy 和附加步骤

journeyPlan 为空的场景

  • TargetPlan.target == null
  • 典型就是 UserGoal.Unknown
  • 这时 RoutePlanner 不会调用 JourneyPlanner.plan()
  • ViewModel 也不会发出 ExecuteJourney，最多只发提示信息

流程图四：从 targetPlan 到 journeyPlan 的联动关系

把两个 planner 串起来看，你会更容易理解为什么同样一句“查看详情”，最后可能得到完全不同的 journey。

Coupling

联动流程

targetPlan 是 journeyPlan 的先决条件。

ParseResult 给出用户意图和解析出的 slot。
→
TargetPlanner 先判断最终目标到底是 Workflow、List、Detail 还是 Dashboard。
→
TargetPlan.target 这一字段直接决定 JourneyPlanner 走哪个 buildXxxJourney 分支。
→
JourneyPlanner 再按当前页和目标页组合出步骤列表。

同一意图，不同命中结果，journey 也会不同

以“打开个人信息详情”为例。

输入 UserGoal.OpenPersonalInfoDetail
→
唯一命中记录 targetPlan.target = DetailTarget
→
journey 分支 列表页路径 + FocusList + OpenFocusedRecordDetail
→
结果 最终会真正打开详情。
输入 UserGoal.OpenPersonalInfoDetail
→
歧义或未命中 targetPlan.target = ListTarget + fallbackMessage
→
journey 分支 列表页路径 + FocusList（可选）+ ShowFallbackMessage + StopAtCurrentScreen
→
结果 不会强开详情，而是停留在列表让用户确认。

三个测试用例，直接验证 PlanningDecision 的生成结果

下面三个案例都来自现有测试文件，它们能直接证明 targetPlan 与 journeyPlan 的解析逻辑。

Tests
Case A

填写个人信息

  1. 输入被构造为 UserGoal.FillPersonalInfo，并带姓名 slot
  2. targetPlan.target 变成 WorkflowTarget
  3. targetPlan.formFields.name 成功合并为“李晨曦”
  4. journeyPlan.steps 先到 SecondFragment，再到表单 fragment

对应测试：planFormGoal_routesThroughSecondThenForm()

Case B

唯一命中详情

  1. 输入被构造为 UserGoal.OpenPersonalInfoDetail + personName = 张雨桐
  2. targetPlan.target 成为 DetailTarget
  3. journeyPlan.steps 里依次出现列表路径、FocusList、OpenFocusedRecordDetail
  4. 说明详情不是直接开的，而是先定位再打开

对应测试：planUniqueDetailGoal_routesThroughListBeforeOpeningDetail()

Case C

详情意图但存在歧义

  1. 输入同样是详情意图，但只给出模糊姓名“张”
  2. targetPlan.target 不再是 DetailTarget，而是回退为 ListTarget
  3. targetPlan.fallbackMessage 不为空
  4. journeyPlan 最后两步是 ShowFallbackMessage 和 StopAtCurrentScreen

对应测试：planAmbiguousDetailGoal_fallsBackToList()

测试本质上已经把答案写出来了： PlanningDecision 不是简单包装对象，而是业务目标解析和路径解析的组合结果。

Index
调用入口

RoutingSessionViewModel.submitUserInput() 负责拿到 PlanningDecision 并回写 UI state。

总调度

RoutePlanner.plan() 负责串联 targetPlanner 与 journeyPlanner。

目标规划

TargetPlanner 与 PersonalInfoRecordResolver 决定 targetPlan 的业务形态。

路径规划

JourneyPlanner 与 EntryActionRegistry 决定 journeyPlan.steps。

最简一句话总结： PlanningDecision = TargetPlanner(ParseResult, currentFields) + JourneyPlanner(currentDestinationId, targetPlan)。 前者决定“去哪里”，后者决定“怎么去”。
