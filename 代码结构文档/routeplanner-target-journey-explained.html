RoutePlanner Analysis

在此工程里，
targetPlanner 与 journeyPlanner
到底是怎么“出来”的

这份说明基于当前工程真实代码整理，重点解释两件事：第一，targetPlanner 和 journeyPlanner 不是从用户输入里临时“解析”出来的，而是在 AppContainer 里被构造并注入到 RoutePlanner；第二，运行时真正被解析的是 ParseResult，随后由 RoutePlanner 分派给两个 planner 分别产出 TargetPlan 和 JourneyPlan。

装配入口 AppContainer 用 lazy 创建依赖，并把两个 planner 传给 RoutePlanner。
运行入口 submitUserInput() 用户文本先解析，再进入 routePlanner.plan(...)。
目标产物 TargetPlan 描述“最终想去哪里”，比如表单、列表、详情、家居看板。
路径产物 JourneyPlan 描述“从当前页怎么过去”，里面是可执行的 JourneyStep 列表。

先澄清一个关键点

如果从字面理解“解析出 RoutePlanner 中的 targetPlanner 和 journeyPlanner”， 容易以为它们是像 ParseResult 一样从文本里解析得到的。实际上不是。

Important

在本工程中，targetPlanner 和 journeyPlanner 是 长期存在的协作对象，由 AppContainer 提前构建； 真正随用户输入变化的是 ParseResult。RoutePlanner 的职责是把 ParseResult 交给 TargetPlanner 生成目标，再交给 JourneyPlanner 生成路径。

静态装配层

关注对象实例如何创建。对应文件主要是 app/src/main/java/com/example/cctest/routing/AppContainer.kt。

  • TargetPlanner(...) 由多个 registry / resolver / engine 组装
  • JourneyPlanner(...) 由 EntryActionRegistry 组装
  • RoutePlanner(targetPlanner, journeyPlanner) 再把二者收口

动态运行层

关注一次用户输入如何被处理。对应入口是 RoutingSessionViewModel.submitUserInput()。

  • 文本先变成 ParseResult
  • TargetPlanner.plan(...) 输出 TargetPlan
  • JourneyPlanner.plan(...) 输出 JourneyPlan

Assembly

装配流程

左边是底层依赖，中间是 planner，右边是最终消费方。

PersonalInfoRepository 提供 mock 个人信息记录数据。
→
PersonalInfoRecordResolver 负责按姓名 / 手机 / 位置 / 城市把 slot 解析成唯一记录或候选集合。
→
TargetPlanner 根据 UserGoal 决定最终目标，并补出 workflow、record、focusRequest 等上下文。
→
RoutePlanner 先调 targetPlanner.plan，再调 journeyPlanner.plan。
→
RoutingSessionViewModel 在 submitUserInput() 里调用 routePlanner.plan(...)。
DestinationContractRegistry 告诉 TargetPlanner 每种目标对应哪类页面与展示名称。
→
TargetPlanner 用 contract 决定 activeTargetLabel、页面类型和目标合法性。
→
RoutePlanner 把 TargetPlan 封成 PlanningDecision。
→
UI State / UiEffect 最终写入 session state，并触发导航或执行 journey。
WorkflowRegistry + WorkflowEngine 为“填写个人信息”提供工作流定义和起始 step。
→
TargetPlanner 生成 WorkflowTarget，并填入 workflowId / currentStepId。
→
RoutePlanner 把工作流目标继续交给 JourneyPlanner 补路径。
→
用户看到表单入口 如果当前页不是目标页，还会先经过第二页。
EntryActionRegistry 保存页面之间的可达边，例如 FIRST -> SECOND -> PERSONAL_INFO_LIST。
→
JourneyPlanner 根据当前页和目标页解析出一串 JourneyStep。
→
RoutePlanner 将 TargetPlan 和 JourneyPlan 一起打包返回。
→
JourneyExecutor 后续执行每一步导航、聚焦列表或打开详情。
可以把 AppContainer 理解成“总装车间”。它不是解析文本，而是把所有零件焊成一套可工作的 routing 子系统。

流程图二：一次用户输入进入 RoutePlanner 的完整路径

这里开始才是运行时解析流程。关键入口位于 RoutingSessionViewModel.submitUserInput()。

Runtime
1

用户输入文本

例如“查看张雨桐详情”或“填写李晨曦的个人信息”。

  • 入口函数：submitUserInput(text, entrySource, currentDestinationId)
  • 先做 trim() 和空串校验
2

组装 ParseRequest

ViewModel 会带上当前页面信息与 session 快照。

  • currentDestinationId 会先转成 AppScreen
  • currentDestination 传给 parser 作为上下文
3

IntentParser 解析

CompositeIntentParser 先走规则解析，必要时再走 LLM。

  • 规则解析命中且置信度够高则直接返回
  • 否则按配置决定是否启用 LLM 或回退规则结果
4

标准化与校验

结果先经过 SlotNormalizer，再经过 ParseResultValidator。

  • 手机号只保留数字
  • 位置必须大于 0
  • 详情意图必须至少带姓名 / 手机 / 位置之一
5

进入 RoutePlanner

routePlanner.plan(result, currentDestinationId, currentFields)

  • 先生成 TargetPlan
  • 再基于 target 生成 JourneyPlan

RoutePlanner 内部调用链

下面这条线正对应 RoutePlanner.plan() 方法体。

输入 ParseResult + currentDestinationId + currentFields
→
targetPlanner.plan(...) 根据 UserGoal 生成 TargetPlan。
→
targetPlan.target 是否为空 为空就不生成 journey；不为空才继续。
→
journeyPlanner.plan(...) 根据当前页与目标页生成 JourneyPlan。
→
PlanningDecision 最终包含 targetPlan 与可选的 journeyPlan。
所以 RoutePlanner 不是一个“大而全”的解析器，它只是把“目标解析”和“路径解析”两个职责顺序组合起来。

流程图三：TargetPlanner 如何把 ParseResult 解析成 TargetPlan

TargetPlanner.plan(result, currentFields) 的核心判断维度是 result.userGoal。也就是先识别“用户要做什么”，再进入不同分支。

Target

TargetPlanner 决策流

这是 when (result.userGoal) 的展开图。

ParseResult.userGoal 四种主要值：FillPersonalInfo、BrowsePersonalInfoList、OpenPersonalInfoDetail、OpenHouseDashboard。
→
FillPersonalInfo 读取表单 workflow，获取首个 step，把当前表单字段与解析出的字段做 merge。
→
输出 WorkflowTarget 得到 workflowId、entryDestinationId、currentStepId = form 和合并后的 formFields。
→
后续含义 目标已经不是“某个单独 fragment”，而是“个人信息工作流的起点”。
ParseResult.userGoal BrowsePersonalInfoList
→
resolveFromSlots(slots) 用姓名 / 手机 / 城市 / 第 N 条记录去解析列表聚焦目标。
→
输出 ListTarget 写入 listFocusRequest、可选 recordRef 和列表提示文案。
→
后续含义 Journey 会导航到列表页，并在必要时执行 FocusList。
ParseResult.userGoal OpenPersonalInfoDetail
→
resolveFromSlots(slots) 先判断能否唯一命中某条记录。
→
唯一命中？ 是：输出 DetailTarget。否：回退成 ListTarget 并附带 fallbackMessage。
→
后续含义 详情不是直接打开 Activity，而是通常先在列表定位，再执行“打开已定位详情”。
ParseResult.userGoal OpenHouseDashboard 或 Unknown
→
OpenHouseDashboard 构造 HouseDashboardLaunchSpec 并输出 HouseDashboardTarget。
→
Unknown 不设置 target，仅返回面向用户的 fallbackMessage。
→
后续含义 没有 target 时，JourneyPlanner 不会被调用。

填表分支的关键动作

  1. 向 DestinationContractRegistry 要 PERSONAL_INFO_FORM contract
  2. 从 WorkflowRegistry.personalInfoWorkflow() 取工作流
  3. 用 WorkflowEngine.start() 拿到第一步 form
  4. 把 currentFields 与解析出的 personalInfoFields 合并
  5. 输出 RouteTarget.WorkflowTarget

列表 / 详情分支为何依赖 RecordResolver

因为文本里给出的“张雨桐”“第 2 条”“13800001001”都不是最终页面对象， 还需要先解析到仓库里的某条记录，才能知道该聚焦哪一条、能不能直接开详情。

  • 唯一命中：生成明确记录引用
  • 多条命中：保留候选并回退列表
  • 零命中：生成模糊查询型 ListFocusRequest

Unknown 分支的真实效果

TargetPlan.target 为 null。这意味着：

  • RoutePlanner 不会进入 journeyPlanner.plan()
  • ViewModel 也不会发出执行路径的 UiEffect.ExecuteJourney
  • 最终只剩提示文案，要求用户换一种说法

RecordResolver 的内部匹配顺序

这段顺序直接决定详情是否唯一、列表是否模糊、以及回退策略。

recordId 如果已知记录 ID，优先级最高，直接精确命中。
→
phone 手机号规范化后做精确匹配；1 条唯一，多条歧义。
→
position 比如“第 2 条记录”，直接按列表序号取仓库中的第 2 条。
→
name + city 先精确姓名，再模糊姓名，城市作为附加过滤条件。
→
city 仅按城市兜底匹配；还不行就返回 none。

流程图四：JourneyPlanner 如何把 TargetPlan 解析成 JourneyPlan

JourneyPlanner 关注的不是“用户想做什么”，而是“从当前页到目标页应该走哪些步骤”。 它的输入是 currentDestinationId 和已经算好的 TargetPlan。

Journey

JourneyPlanner 总流程

这一段对应 JourneyPlanner.plan() 主体逻辑。

读取 targetPlan.target 如果目标为空，直接返回 null。
→
currentDestinationId -> AppScreen 通过 AppScreen.fromDestinationId() 把当前 fragment id 映射成屏幕枚举。
→
normalizeEntryScreen() UNKNOWN 会被强制视为 SECOND，避免路径规划失去起点。
→
按 target 类型分派 WorkflowTarget / ListTarget / DetailTarget / HouseDashboardTarget / NavTarget

resolvePathSteps() 的核心规则

  • 如果 entryScreen == targetScreen，返回空步骤，表示已经在目标屏幕
  • 否则向 EntryActionRegistry.resolvePath(from, to) 要一条可达路径
  • 如果拿到路径，就把每个 EntryAction 翻译成一个或多个 JourneyStep
  • 如果拿不到路径，就退回到传入的 directStep

EntryActionRegistry.resolvePath() 为什么稳定

它内部是一个宽度优先搜索。也就是说，它不是写死某一条路径，而是从当前页面出发， 沿着 action 图找出第一条最短可达路径。

  • FIRST -> SECOND
  • SECOND -> PERSONAL_INFO_LIST
  • SECOND -> PERSONAL_INFO_FORM
  • PERSONAL_INFO_LIST -> PERSONAL_INFO_DETAIL
WorkflowTarget
→
目标屏幕视为 PERSONAL_INFO_FORM，步骤来自路径规划；如果找不到路，就直接用“进入个人信息工作流”这一步。
ListTarget
→
先到 PERSONAL_INFO_LIST，如果有 listFocusRequest 再追加 FocusList；如果有回退文案，再追加 ShowFallbackMessage；最后一定追加 StopAtCurrentScreen。
DetailTarget
→
先走到 PERSONAL_INFO_LIST，然后生成或复用 focusRequest，追加 FocusList，最后再追加 OpenFocusedRecordDetail。
HouseDashboardTarget
→
目标屏幕视为 HOUSE_DASHBOARD，最终步骤里会出现 OpenHouseDashboard。
NavTarget
→
不做路径搜索，直接产出一个 NavigateToFragment。

典型路径例子：从 FIRST 打开个人信息详情

这也是测试里最典型的一条链路。

当前页 FIRST
→
BFS 找路 FIRST -> SECOND -> PERSONAL_INFO_LIST
→
路径步骤 NavigateToFragment(SecondFragment) + NavigateToFragment(PersonalInfoListFragment)
→
补业务步骤 FocusList + OpenFocusedRecordDetail
→
最终 JourneyPlan 一条完整可执行步骤链，供后续 executor 消费。

三个真实案例，把两层 planner 串起来看

这里选用仓库中的 mock 数据和现有测试场景，帮助你把抽象逻辑落到具体输入上。

Cases
Case A

“填写李晨曦的个人信息”

  1. 规则解析命中 FillPersonalInfo，并抽出 personalInfoFields.name = 李晨曦
  2. TargetPlanner 读取个人信息 workflow，拿到首步 personalInfoFormStepFragment
  3. 当前表单值与解析值做 merge，形成新的 formFields
  4. 产出 WorkflowTarget
  5. JourneyPlanner 若发现当前在 FirstFragment，会先走到 SecondFragment，再进表单 fragment

这正对应 RoutePlannerTest.planFormGoal_routesThroughSecondThenForm。

Case B

“查看张雨桐详情”

  1. 规则解析命中 OpenPersonalInfoDetail，slot 里有 personName = 张雨桐
  2. PersonalInfoRecordResolver 在仓库中唯一命中记录 张雨桐
  3. TargetPlanner 产出 DetailTarget，并顺手带上一个 listFocusRequest
  4. JourneyPlanner 先规划到列表页，再追加 FocusList 和 OpenFocusedRecordDetail
  5. 最终执行顺序通常是：第二页 -> 列表页 -> 定位记录 -> 打开详情

这正对应 RoutePlannerTest.planUniqueDetailGoal_routesThroughListBeforeOpeningDetail。

Case C

“查看张的详情”

  1. 规则解析仍可能命中 OpenPersonalInfoDetail
  2. 但 RecordResolver 用模糊姓名“张”会得到多条候选，无法唯一确定
  3. TargetPlanner 不再生成 DetailTarget，而是回退生成 ListTarget
  4. 同时填入 fallbackMessage，提示“详情目标存在歧义，已回退到列表页供你确认”
  5. JourneyPlanner 会在列表相关步骤后追加 ShowFallbackMessage 与 StopAtCurrentScreen

这正对应 RoutePlannerTest.planAmbiguousDetailGoal_fallsBackToList。

这三个案例能看出两个 planner 的分工边界： TargetPlanner 决定“去哪儿”， JourneyPlanner 决定“怎么去”。

你可以这样理解整个架构

下面这一版是最适合脑内记忆的压缩结论。

Summary
1. 装配

AppContainer 构造好 targetPlanner、journeyPlanner 和 routePlanner。

2. 文本解析

用户文本先变成标准化、校验过的 ParseResult。

3. 目标规划

TargetPlanner 把意图和 slot 变成 TargetPlan。

4. 路径规划

JourneyPlanner 再把 TargetPlan 变成可执行的 JourneyPlan。

换句话说，RoutePlanner 本身几乎不“懂业务细节”，它更像一个总调度器。 真正的业务判断主要在 TargetPlanner，真正的路径组合主要在 JourneyPlanner。
